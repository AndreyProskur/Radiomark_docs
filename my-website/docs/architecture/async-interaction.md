# Асинхронное взаимодействие

В платформе Radiomark асинхронное взаимодействие применено для надёжной доставки финального документа пациенту. Такой подход разгружает основной поток обработки и гарантирует, что уведомления дойдут даже при кратковременных сбоях.

## Поток событий

1.  **Модератор** проверяет и публикует медицинский снимок.
2.  Система отправляет событие `StudyApprovedEvent` в брокер сообщений.
3.  **Врач**, назначенный на соответствующую заявку, получает уведомление о завершении модерации.
4.  Врач формирует финальный документ: генерирует PDF с заключением и подписывает его электронной подписью.
5.  Сразу после подписания система публикует второе событие – `MedicalDocument`.
6.  Это событие инициирует отправку готового документа пациенту (по электронной почте или в личный кабинет).

## Выбор технологии: Apache Kafka

Для реализации событийной коммуникации выбран **Apache Kafka** – распределённый брокер сообщений, оптимизированный для высоконагруженных систем.

| Критерий | Обоснование |
|:---------|:------------|
| **Порядок сообщений** | События по одной заявке должны обрабатываться строго последовательно. Kafka сохраняет порядок в пределах партиции, что исключает путаницу у пациента. |
| **Надёжная доставка** | Механизмы репликации и офсетов гарантируют, что события не потеряются даже при падении консьюмера – после восстановления он продолжит чтение с нужного места. |
| **Масштабируемость** | С ростом базы снимков и количества заявок нагрузка будет расти. Kafka горизонтально масштабируется, позволяя обрабатывать миллионы сообщений без деградации. |
| **Батчевые операции** | Позволяют накапливать сообщения и обрабатывать их пакетами, снижая нагрузку на базу данных. |
| **Интеграция с Object Storage** | В теле события передаётся только ссылка на PDF-файл, само содержимое хранится в S3/MinIO. Это экономит сетевые ресурсы и упрощает хранение больших файлов. |

## Контракт события MedicalDocument (Avro)

Событие `MedicalDocument` генерируется после подписания документа врачом. Оно содержит ссылку на файл и метаданные для идентификации заявки и врача.

<details>
<summary>📄 Показать схему Avro</summary>
```json
{
    "type": "record",
    "name": "MedicalDocument",
    "namespace": "com.radiomark.events",
    "doc": "Событие готовности медицинского документа для пациента",
    "fields": [
        {
            "name": "documentId",
            "type": "string",
            "doc": "ID документа"
        },
        {
            "name": "caseId",
            "type": "string",
            "doc": "ID заявки пациента, к которой относится документ"
        },
        {
            "name": "doctorId",
            "type": "string",
            "doc": "ID врача, создавшего и подписавшего документ"
        },
        {
            "name": "documentUrl",
            "type": "string",
            "doc": "Ссылка на PDF-файл заключения"
        },
        {
            "name": "signedAt",
            "type": "long",
            "logicalType": "timestamp-millis",
            "doc": "Время подписания документа"
        }
    ]
}
```
</details>

:::tip[Почему timestamp-millis?]

Поле `signedAt` использует тип `long` с логической аннотацией `timestamp-millis`. Это стандартный подход для передачи времени в Avro‑сообщениях Kafka:

- **Компактность.** Целое число занимает 8 байт против ~20 байт у строки ISO 8601.
- **Однозначность.** Unix‑время в миллисекундах не зависит от часовых поясов.
- **Совместимость.** Kafka Streams и другие инструменты экосистемы понимают этот тип из коробки.

:::
